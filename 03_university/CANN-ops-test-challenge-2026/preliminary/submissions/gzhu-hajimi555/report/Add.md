# Add 算子测试报告

## 一、算子理解
Add 算子用于加法运算，不仅包含基础的张量相加，还引入了缩放因子 $\alpha$ 以及支持标量与张量之间的混合运算。其数学定义及 API 形态可抽象为：
* **Add (Tensor-Tensor)**: $out=self+\alpha \times other$
* **Adds (Tensor-Scalar)**: $out=self+\alpha \times other\_scalar$
* **AddV3 (Scalar-Tensor)**: $out=self\_scalar+\alpha \times other$

Add 算子的工程难点与测试风险主要体现在：
* **数据类型的广度与混合精度**：需要支持包括常规浮点、量化整型、复数类型以及混合精度（如 FP16 与 FLOAT 混合）的运算。
* **广播机制的复杂性**：涉及多维度的广播（如 4x1 与 1x4 广播至 4x4，3D 广播等）。
* **缩放因子 $\alpha$ 的传播机制**：$\alpha$ 可作为不同精度的标量乘数（MulAdd/Axpy 特性），对溢出和精度损失极为敏感。

* **Tensor-Scalar 场景**：`aclnnAdds` 与 `aclnnInplaceAdds` 接口测试。
* **Scalar-Tensor 场景**：`aclnnAddV3` 与 `aclnnInplaceAddV3` 接口测试。

涵盖了各种边界与常规形状的测试用例：

**3. 数据类型 (dtype) 与 $\alpha$ 特性全面覆盖**
* **广泛的 dtype 支持**：覆盖 FLOAT, FLOAT16, BF16, INT32, INT8, UINT8, INT64, BOOL, INT16, DOUBLE, COMPLEX64, COMPLEX128。

代码通过 `ReportErrorCase` 宏集中验证了三类 API 的防错处理机制：
* 空指针（`nullptr`）拦截：涵盖 `self`, `other`, `alpha`, `out` 为空的场景。

* 不支持的维度与广播失败：维度超过 8 维的拦截、输出 `shape` 大小不匹配等。

## 四、精度分析
Add 的精度稳定性考量主要基于不同底层精度实现的转换损耗。

### 场景一：$\alpha$ 缩放与 Axpy 运算
* **策略**：使用极值如 `1e6`、负值 `-2.5` 以及浮点 $\alpha$ 应用于整数类型的乘加测试。
* **结论**：测试设计明确区分了 $\alpha$ 为 $1$（纯加法）与不为 $1$（MulAdd）的计算路径，确保缩放因子的数值稳定性机制正确被触发。

### 场景二：半精度与混合精度支持
* **策略**：针对 FLOAT16 与 BFloat16，引入了诸如 `0x3C00`, `0x3F80` 等 16 位整数模拟内存数据，并实施 FLOAT16+FLOAT 混合精度计算校验。
* **结论**：精度对齐机制符合设计规范，隐式类型推导在半精度向单精度转换时表现符合预期。

### 场景三：双精度与复数计算
* **策略**：提供了专门针对 `COMPLEX64`/`COMPLEX128` 和 `DOUBLE` 类型的测试组合。
* **结论**：在 AICPU 侧和计算图架构上验证了高精度浮点格式以及复数实虚部独立累加的准确性。

## 五、反思与改进
1.  **大规模性能压力测试空间**：当前测试张量最大规模为 1024 元素。在功能正确性已保证的前提下，可补充 GB 级别的极限性能压测，以暴露超大显存调度下的 DMA 传输瓶颈。
2.  **精度校验标准的严谨性**：当前宏 `CHECK_RET` 验证的是 ACL 执行结果是否为 `ACL_SUCCESS`，缺乏将设备端结果取回并与 CPU 标准数学库严格比较的容差（Tolerance）匹配逻辑机制。
3.  **异常用例的环境强健度**：虽然已针对 `UINT32` 等设置了类型排斥用例，但在系统资源耗尽（如 `aclrtMalloc` 失败触发 `OOM`）的被动异常注入上仍属空白。
