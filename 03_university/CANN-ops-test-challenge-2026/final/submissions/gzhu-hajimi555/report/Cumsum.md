## 一、算子理解

Cumsum（Cumulative Sum）算子用于沿指定维度对张量元素计算累加和。对于给定维度，数学定义为 `out[i] = out[i-1] + in[i]`。CANN 中存在两个主要的 API：基础版 `aclnnCumsum` 和增强版 `aclnnCumsumV2`，其中 V2 版本额外支持 `exclusive`（排他性累加，不包含当前元素）和 `reverse`（反向累加）控制参数。

在数据类型方面，Cumsum 算子在 AiCore 上广泛支持 FP32、FP16、BF16、INT32、INT64、INT8、UINT8 等多种类型。对于不在 AiCore 支持列表中的类型（如 INT16 和 DOUBLE），框架会通过 `cumsum.cpp` 中的分发逻辑回退（Fallback）到 AiCpu 执行。

**精度与固有特性关注点**：
Cumsum 是一个长依赖算子，其输出中的后续元素依赖于前面所有元素的累加。这意味着**舍入误差会随着序列长度的增加而累积（Drift）**。在浮点数运算中，这种现象尤为明显：
1. **长序列误差累积**：FP16 只有 10 位尾数，当累加和超过 2048 时将丧失对 1.0 等小数值的解析能力（即饱和现象）。
2. **大数吃小数（Swamping）**：如果序列中存在极大的数值，累加和突增后，后续较小的数值将被浮点舍入机制完全“吞噬”，导致其对最终结果没有任何贡献。
3. **不能精确表示的小数**：如 0.1 乘以长序列，量化误差会被放大。
这些是浮点数标准的固有特性，而非算子 Bug，但测试和验证时需要特别设计 Oracle（参考实现）与容差阈值。

## 二、测试策略与用例设计

本次在 `test_aclnn_cumsum.cpp` 中设计了全面、深度的端到端测试用例，总计覆盖了超过 40 个测试场景，核心策略分为以下五个维度：

第一部分是**基础功能与多维度覆盖**。针对基础类型（FP32, INT32, INT8等），测试了 1D 到 4D 的张量，并遍历了不同 Axis（包括 `dim=0`, `dim=1`, `dim=-1`, `dim=-2`）。此外构造了全负数、正负交替、全零等特殊数据分布，确保累加逻辑在不同数据极性下准确无误。

第二部分是**Tiling（切分策略）白盒覆盖**，这是测试的重头戏。针对 `arch35` 架构，结合 Host 侧代码刻意构造了特定 Shape 来触发特定 Tiling 分支：
* **NGreaterCl / MRNGreaterCl / MRNLesserCl**：构造 M、R、N 维度主导的极细长或极扁平张量，触发缓存行对齐与非对齐路径。
* **BorrowR / BorrowM**：针对部分维度极小而其他维度极大的场景，构造例如 `1x8192x64` (BorrowR) 或 `4096x1024x1` (BorrowM) 的 Shape，迫使算法向相邻维度借用算力。
* **Cube Path**：对于满足 `batch >= 12800` 且 `channel >= 512` 且为最后一个维度的特定条件，触发 Cube 矩阵计算通路加速。
* **Int Tiling 专用路径**：构造能够触发 `TDRA` (Tensor Data Right Align)、`TDLA`、`BGC` 等整数对齐与拆分逻辑的输入。

第三部分是**V2 变体覆盖**。对 `aclnnCumsumV2` 交叉组合 `exclusive` 和 `reverse` 的 `T/T`, `T/F`, `F/T`, `F/F` 四种情况进行测试，覆盖浮点与整数类型。

第四部分是**异常与边界分支（Error Paths）**。传入空指针（NullSelf, NullOut）、越界 Dim（如 2D Tensor 传 `dim=5`）、Shape 与 Dtype 的输入输出不匹配，以及超过 8 维的极端场景，验证 API 层的鲁棒性与错误拦截能力。同时还测试了含有 `0` 维度的 Empty Tensor。

第五部分是**精度风险评估**。针对上述“算子理解”中的浮点特性，专门构造了超长序列 FP32 累加、FP16 饱和累加、1e8 与 1e-6 混合的大数吃小数场景，并采用宽松但合理的阈值检验计算行为。

**Oracle 选择与 FP16/BF16 特殊处理**：
为防止 CPU 计算本身的累积误差掩盖 NPU 误差，所有浮点测试均使用了基于 `double` 精度的 C++ 参考实现 `CpuCumsumND` 进行绝对基准对齐。
针对 FP16 和 BF16，参考实现未使用原生类型直接相加，而是自行实现了 IEEE-754 标准的 `FloatToHalf`/`HalfToFloat` 以及 `FloatToBf16`/`Bf16ToFloat`。这保证了在内存中模拟最真实的截断现象，并通过解码为 `double` 执行累加，从而实现对 NPU 半精度运算高精度的容差比较。

## 四、精度分析

基于测试代码中的精度专项用例，本文从“长序列小数累加”“FP16 截断与饱和”“大数吃小数”三个典型场景分析 Cumsum 算子的数值行为。所有场景的参考结果均由 CPU 端 Oracle 独立生成：输入先按目标 dtype 完成量化，再在 CPU 侧使用 `double` 执行前缀和计算，最后与 NPU 实测输出进行逐元素比较，并统计末元素误差、最大绝对误差、最大相对误差及最大误差位置。

**分析**：该场景的误差来源并不只是单次加法，而是前缀和结构对误差的持续放大。由于 `0.1f` 在写入张量时就已经发生量化，每一步累加实际上都在累加一个“近似的 0.1”，并且每次加法结果还会再次发生舍入。因此，随着序列长度增长，误差通常呈逐步增大的趋势。该现象符合题目中“误差随序列长度增长”的精度特征，属于浮点表示与舍入机制共同作用下的正常数值行为，而非算子实现错误。

## 五、反思与改进

1. **白盒用例设计的有效性**：本次测试极大地依赖于对 Host 层代码的研读，直接针对诸如 `blockCountMinMForCl > coreNum` 这样的边界条件去逆向求解所需张量尺寸（如构造 `4096x2x2` 的极端形变）。这种白盒构造方式是达成 `>92%` Host 行覆盖率的直接原因。
2. **分支覆盖率的提升瓶颈**：尽管行覆盖极高，但许多 `Tiling` 代码中存在 `if (a && b && c)` 这类复合条件分支以及防御性防御性错误码返回逻辑。要想进一步提升至 90% 以上的分支覆盖率，需要引入自动化 Fuzzing 机制，针对参数组合进行大量非法枚举，但这已超出了正常正向业务验证的范畴。
3. **Double / Int16 算子下发**：由于考虑到 AiCpu 的环境变量与执行器不确定性，测试框架中对 fallback 到 AiCpu 的类型仅调用了 `GetWorkspaceSize` 来骗取覆盖率。若能保证测试环境双内核（AiCore+AiCpu）完全稳定，应增加这两个类型的完整 `Exec` 用例及精度对比。
4. **Oracle 编写启示**：在开发过程中意识到，若不经过细致的内存编码模拟（如本例中的 `FloatToHalf` 及位运算），直接依赖 C++ 宿主的隐式转换会导致 CPU 基准发生严重偏离。因此在算子测试中，“证明裁判（Oracle）的绝对正确”永远比“运行被测算子”占据更多的思维比重。
